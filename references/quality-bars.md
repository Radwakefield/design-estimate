# Quality bars

Reading lenses + gates for each tier: the **one-pager engagement model** for the SWAG tier, and
the **Design Kick-off readiness checklist** for the Refined (PRD) tier. Both calibrate confidence
and name the review/crit steps.

## One-pager engagement model (SWAG reading lens)
Source: Design Playbooks — "Engagement Model: One-Pagers". How Product Design shapes a one-pager
and what a solid one looks like. Use it when reading a one-pager in the SWAG tier: each section
maps onto a sizing dimension, and a 3-item checklist calibrates confidence.

A one-pager moves through: **Identify the problem → Ideate solutions → Set goals → Build confidence.**

| One-pager section | What design shapes (artifacts) | Maps to dimension |
|---|---|---|
| **Problem statement** | Frame the problem through a user lens; validate against existing research + UX data (journey maps, support tickets, research synthesis) | **Discovery** |
| **Solution** | Participate in idea generation; reference existing flows/patterns; offer heuristics/best practices (past designs, experience heuristics) | **Scope** |
| **Success criteria** | Size impact on product OKRs *with PM*; give UX rationale for the projections (UXR insights inform PM sizing) | **Impact** |
| **Confidence level** | Surface UX risks/unknowns; assess need for validation; name design/research gaps (risk assessment, research-gap checklist, concept-test plan) | **Discovery / risk** |

**Readiness checklist** (each miss → lower confidence + a targeted question + a flagged risk; it does
NOT change the size):
1. The user problem is **grounded in evidence or research** (not asserted). Missing → Discovery may be
   higher than the doc implies; ask what evidence exists.
2. **Success criteria link UX outcomes to OKRs** (not placeholder `+x%`). Missing → ask for the baseline;
   flag impact as unquantified.
3. The **Confidence section names experience risks or research gaps**. Missing → the doc hasn't surfaced
   its own unknowns; treat ambiguity as higher.

Engagement is *early* — one-pager work starts the moment a problem is identified, so a SWAG read is
inherently low-to-moderate confidence; the checklist is how you say *why*.

## Design Kick-off readiness checklist (PRD gate)
A PRD may pass or intentionally not pass - what matters is the choice is explicit.
1. **Problem Framing** - no solution bias; coherent, believable framing.
2. **Understanding & Evidence** - directional soundness; root-cause analysis; contextual
   credibility (user needs + business constraints).
3. **Impact & Scope** - singular focus; success metrics (baseline + target); transparency
   on risks/tradeoffs.
4. **Execution Readiness** - constraint mapping; dependency check; visibility of unknowns.

In the Refined tier: failed checklist items become flagged risks and can lower confidence.

## Design review expectations
Reviews are structured alignment moments, not walkthroughs. Every designer must: articulate
the problem; summarize data/constraints; present in themes/decisions; make tradeoffs visible;
end with a clear decision/ask. Minimum attendees: Product, Design, Engineering leadership.

Stage-specific:
- **Early-phase (divergent) & Pre-UXR:** breadth over polish; multiple directions; explicit
  about what's being tested; prototypes with correct entry points + complete journeys.
- **Mid-phase:** show evolution from prior feedback; frame specific tradeoffs.
- **Final / Pre-Engineering:** precision + completeness; current vs proposed; all states +
  edge cases; system implications; rollout considerations.

Number of review/crit steps scales with size - see sizing-matrix.md "Feedback cycles".
